# Easy Linux · Network Monitoring · IDOR · Linux Capabilities
Difficulty Easy
OS Linux
User Flag ✓ Captured
Root Flag ✓ Captured
Target IP 10.129.28.91

Cap is an Easy-rated Linux machine hosting a network monitoring web application. The attack chain begins with an Insecure Direct Object Reference (IDOR) vulnerability — iterating a numeric URL parameter exposes another user's PCAP containing FTP credentials transmitted in plaintext. Those credentials are reused on SSH, granting user access. Privilege escalation abuses a misconfigured Linux capability (cap_setuid) assigned to the Python 3.8 interpreter to spawn a root shell with a single one-liner.

IDOR — Web App → PCAP Download → FTP Cleartext Creds → SSH (nathan) → cap_setuid — Python 3.8 → ROOT

02
Enumeration Web Application — Port 80

Navigating to http://10.129.28.141 presents a network monitoring dashboard. The current user nathan is already authenticated. The application exposes a "Security Snapshot" feature that captures a burst of live network traffic and generates a downloadable .pcap file.

After triggering a snapshot, the application redirects to the following URL where the capture file can be downloaded:

http://10.129.28.141/data/1
POC 01 — Security Snapshot Redirect
Dashboard redirects to /data/<id> after triggering a snapshot capture.

The URL follows the pattern /data/<id> where the trailing integer is a simple auto-incremented counter. No session token, CSRF nonce, or user-specific component is embedded in the resource path. This immediately flags an IDOR risk — decrementing the ID to 0 should access a resource created prior to the current session.

⚑ Analyst Note
Sequential Integer ID — Classic IDOR Setup
The application uses a monotonically increasing integer as the sole identifier for PCAP resources. With no authorisation enforced at the object level, any value for <id> is reachable. ID 0 likely predates any user session and belongs to system-level or administrative traffic — exactly where sensitive credentials are most likely to appear.
03
Exploitation IDOR → PCAP Analysis → Cleartext FTP Credentials

IDOR — Accessing Another User's PCAP

⚠ Vulnerability CWE-639 · OWASP API01:2023
Insecure Direct Object Reference (IDOR)
The /data/<id> endpoint serves PCAP files using only the integer id as a lookup key. The server performs no ownership check — it does not verify that the authenticated session is entitled to access the requested resource. Any authenticated user can enumerate all PCAP files by iterating the ID parameter, constituting a broken object-level authorisation (BOLA) vulnerability.

Manually modifying the URL from /data/1 to /data/0 causes the server to serve a PCAP belonging to another user's session without any authorisation challenge:

http://10.129.28.141/data/0
POC 02 — IDOR Confirmed
PCAP belonging to another session is served with no authorisation check at /data/0.

The PCAP file downloads automatically. No error, no redirect, no access denied — the vulnerability is confirmed. The server blindly returns whichever resource maps to the provided integer, regardless of the requesting session's identity.

⚠ Security Note
Credential Reuse Across Services
The same password used for FTP — a cleartext protocol — is reused for SSH. Password reuse means a single point of exposure (the FTP PCAP in this case) cascades into full shell access on a far more critical service. Services should always be provisioned with independent credentials, and FTP should never be used where SSH/SFTP is available.

POC 06 — LinPEAS Capabilities Finding
LinPEAS flags /usr/bin/python3.8 under the Files with Capabilities section.
⚠ Vulnerability Linux Capability Abuse
cap_setuid Assigned to Python 3.8
The cap_setuid capability allows a process to arbitrarily change its effective user ID — including to UID 0 (root) — without requiring a setuid binary or sudo entry. Assigning this to a general-purpose interpreter like Python is functionally equivalent to granting unrestricted root access to any user who can execute it. Since nathan can run /usr/bin/python3.8, privilege escalation is trivial.

Capability Verification & Root Shell

cap_setuid Exploitation — Root Shell
# Verify the capability is set
nathan@cap:~$ getcap /usr/bin/python3.8
/usr/bin/python3.8 = cap_setuid+eip
# Exploit: set UID to 0, then exec bash
nathan@cap:~$ /usr/bin/python3.8 -c 'import os; os.setuid(0); os.execl("/bin/bash", "bash")'
root@cap:~# id
uid=0(root) gid=1001(nathan) groups=1001(nathan)
root@cap:~# whoami
root
POC 07 — Capability Verified & Root Achieved
getcap confirms cap_setuid; the Python one-liner yields a root shell.

The one-liner breaks down as follows:

os.setuid(0) → Sets the effective UID of the Python process to 0 (root)
os.execl("/bin/bash","bash") → Replaces the current process with /bin/bash, inheriting UID 0
⚠ Defence Note
Never Assign cap_setuid to Interpreters
Capabilities such as cap_setuid, cap_setgid, and cap_sys_admin should only be granted to purpose-built, minimal binaries where no other design is feasible. Assigning them to scripting interpreters (Python, Ruby, Perl, Node.js) provides zero security boundary because the interpreter can run arbitrary code. Audit capability assignments with getcap -r / 2>/dev/null during any hardening review.

The complete attack chain on Cap demonstrates three compounding weaknesses: a web application's failure to enforce object-level authorisation (IDOR) leading to PCAP access; an insecure protocol exposing credentials in cleartext (FTP); credential reuse across services; and finally a Linux capability misconfiguration (cap_setuid) that converts user-level access into full root in a single command. Each flaw is individually low-complexity but together they form a complete, uninterrupted compromise path.

POC 09 — HTB Achievement
Cap marked as fully owned on Hack The Box.
★ Key Takeaways
Lessons from Cap
Enforce object-level authorisation on every API endpoint — never trust client-supplied IDs alone. Eliminate FTP in favour of SFTP or SCP wherever remote file transfer is required. Enforce unique, service-specific credentials for every service account. Audit Linux capability assignments regularly; interpreters and shells must never carry elevated capabilities.
