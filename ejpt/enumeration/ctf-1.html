# SMB Enum · SMB Brute-Force · FTP Non-Standard Port · SSH Banner Inspection

This lab focuses on active enumeration methodology against a single Linux target. The exercise covers the complete service-interrogation workflow: null-session SMB share enumeration with wordlist iteration, SMB credential brute-forcing to access a protected user share, FTP credential brute-forcing on a non-standard port identified only through a full -p- TCP scan, and SSH login banner inspection to retrieve a flag without requiring a shell. Four flags are captured across four discrete attack paths — each exploiting a distinct misconfiguration.

Full -p- TCP Scan → SMB Null Session → Flag 1 → SMB Brute-Force (josh) → Flag 2 → Hydra FTP :5554 (alice) → Flag 3 → SSH Login Banner → Flag 4
01
Reconnaissance Full TCP Port Scan → Service Fingerprinting

The first step is a full SYN scan across all 65,535 TCP ports. A default Nmap scan covers only the top 1,000 most common ports — running a restricted scan here would have missed the FTP service on port 5554 entirely, eliminating Flag 3 from the attack surface before enumeration even began.

⚠ Methodology Note
Always Run a Full -p- Scan First
Default Nmap scans silently omit anything outside the top 1,000 ports. Non-standard port placement is a common low-effort attempt at security through obscurity. The additional time cost of -p- is trivial compared to the risk of missing an entire service. Run it first, then focus version and script scans on the confirmed open ports only.
Phase 1 — Full SYN Scan
$ nmap -sS -p- -T4 target.ine.local
POC 01 — Nmap Full Port Scan
Full SYN scan identifies all open ports including FTP on the non-standard port 5554.

With open ports confirmed, a version and script scan is focused against those specific ports to fingerprint service versions and pull useful NSE script output:

Phase 2 — Service Version & Script Scan
$ nmap -sV -sC -p 22,139,445,5554 target.ine.local
PORT STATE SERVICE VERSION
22/tcp open ssh OpenSSH
139/tcp open netbios-ssn Samba smbd
445/tcp open netbios-ssn Samba smbd
5554/tcp open ftp vsftpd (non-standard port)
Port Proto Service Notes
22 tcp SSH Banner / MOTD inspection — Flag 4
139 / 445 tcp SMB / Samba Null session enum — Flag 1 & 2
5554 tcp FTP (non-standard) Invisible to default Nmap — Flag 3. Only found via -p-
02
SMB Enumeration Null Session Share Listing → Wordlist Iteration

With Samba confirmed on 139/445, the correct opening move is a null-session share listing — no credentials required. This frequently returns the full list of shares, user policy information, and in misconfigured environments, directory contents directly.

smbclient — Null Session Share Listing
$ smbclient -L //target.ine.local -N

Automated Share Iteration via Wordlist

Rather than checking shares manually one at a time, a Bash loop iterates the full share wordlist, attempting a null-session directory listing against each potential share name. Shares that permit anonymous access return file listings; those that don't return an access denied error. This approach is faster, more thorough, and requires no guessing.

Wordlist Path
/root/Desktop/wordlists/shares.txt
Bash Loop — SMB Share Enumeration
$ while read share; do
> echo "shareName = $share"
> smbclient \\\\target.ine.local\\$share -N -c 'ls'
> done < /root/Desktop/wordlists/shares.txt
# Output — anonymous shares return file listings
# Access-denied shares are immediately apparent from error output
⚑ Technique Note
Why Loop Rather Than Check Manually
Manual share checking requires a separate command per share and relies on the analyst to interpret which names are worth trying. The wordlist loop removes both bottlenecks — it is exhaustive, systematic, and produces a clear binary result (listing vs. access denied) for every candidate share name in seconds. This is the correct default approach against any SMB target.
⚑ 01
Flag 1 — Anonymous SMB Share Access Null-session smbclient → file retrieval

The wordlist iteration reveals a share that accepts anonymous (null-session) connections and exposes its contents without any credential prompt. Connecting directly and retrieving flag1.txt requires no password.

Anonymous SMB Access — Flag 1 Retrieval

POC 02 — Flag 1 Captured
Anonymous smbclient session listing flag1.txt and printing its contents.
Flag 1 Anonymous SMB Share → flag1.txt
Method: Null-session smbclient — no credentials required
⚠ Misconfiguration CWE-284 · Improper Access Control
Anonymous SMB Share Access
Null-session access means any unauthenticated client on the network can list and read files in the share. No credentials are required — the share is effectively public. In a real engagement this is a critical finding: all files in the share are exposed to any host on the network segment, including sensitive documents, configuration files, or credentials.
⚑ 02
Flag 2 — Weak Password on Private SMB Share SMB Credential Brute-Force → Authenticated Access (josh)

The second flag is behind a user's private share protected by a password. The lab hint confirms a user named josh has a weak password. Private shares in SMB environments are frequently named after their owning user — knowing the username makes the share name trivially predictable. The attack reduces to a single credential brute-force.

SMB Credential Brute-Force — josh
$ cat /root/Desktop/wordlists/shares.txt
# wordlist confirms share name: josh
# Brute-force josh's SMB credentials
$ hydra -l josh \
-P /root/Desktop/wordlists/unix_passwords.txt \
smb://target.ine.local \
-V
[445][smb] host: target.ine.local login: josh password: <password>

Authenticated SMB Access — Retrieve Flag 2

With josh's password recovered, connect to the private share directly using authenticated credentials and retrieve the flag:

Authenticated SMB Session — Flag 2 Retrieval
$ smbclient \\\\target.ine.local\\josh -U josh
Password for [WORKGROUP\josh]: ***
Try "help" to get a list of possible commands.
smb: \> dir
. D 0
.. D 0
flag2.txt N 32
smb: \> get flag2.txt
getting file \flag2.txt of size 32
smb: \> exit
$ cat flag2.txt
POC 03 — Flag 2 Captured
Authenticated smbclient session as josh, retrieving flag2.txt from the private share.
Flag 2 josh's SMB Share → flag2.txt
Method: SMB credential brute-force → authenticated smbclient access
⚠ Misconfiguration Weak Credentials
User Share Named After Account + Weak Password
Predictable share naming (share = username) means that once a valid username is known — from LDAP, RPC, or prior enumeration — the target share name requires zero additional guessing. Weak passwords on personal shares are a common finding in SMB environments. Always brute-force user accounts on Samba after confirming null-session access fails — the two attack paths are complementary, not exclusive.
⚑ 03
Flag 3 — FTP Brute-Force on Non-Standard Port Hydra -s 5554 → alice → FTP login

The full -p- scan identified FTP running on port 5554 rather than the standard port 21. A default Nmap scan would have missed this entirely. The lab hint names the user as alice. Hydra's -s flag redirects the brute-force to the non-standard port — moving a service to an obscure port provides zero additional security once the port is identified.

Hydra — FTP Brute-Force on Port 5554
$ hydra -l alice \
-P /root/Desktop/wordlists/unix_passwords.txt \
ftp://target.ine.local \
-s 5554 \
-t 16 \
-V
[5554][ftp] host: target.ine.local login: alice password: pretty
-l alice → Single username target
-P <list> → Iterate unix_passwords.txt wordlist against the account
-s 5554 → Override default port — attack the non-standard FTP port directly
-t 16 → 16 parallel threads — faster throughput without triggering rate limits
-V → Verbose — print each attempt for visibility
Username alice
Password pretty
Service FTP
Port 5554 (non-standard)

FTP Login → Flag 3 Retrieval

POC 04 — Flag 3 Captured
Hydra brute-force result and FTP session as alice on non-standard port 5554.
Flag 3 FTP :5554 (alice) → flag3.txt
Method: Full -p- scan → Hydra -s 5554 → authenticated FTP
⚠ Misconfiguration
Non-Standard Port + Weak FTP Credentials
Port obfuscation is not a security control — it is defeated by a single Nmap -p- flag. Once the port is identified, Hydra's -s option trivially redirects the attack. FTP transmits credentials in cleartext on top of this. In a real environment, FTP on any port should be replaced with SFTP, and accounts must have strong, unique passwords not present in any wordlist.
⚑ 04
Flag 4 — SSH Login Banner (MOTD) ssh admin@target → Banner / MOTD Inspection

The fourth flag's hint describes "a warning meant to deter unauthorised users from logging in" — the textbook description of an SSH pre-login banner or post-authentication MOTD. These are defined in /etc/issue.net and /etc/motd respectively, and are displayed automatically to every connecting client. No filesystem navigation or privilege escalation is required — the flag surfaces on login.

SSH Login — Banner Inspection
$ ssh admin@target.ine.local
# Flag appears automatically in the login banner / MOTD
# before or immediately after authentication completes
##############################################
# WARNING: Authorised Access Only #
# FLAG: <flag4_value> #
##############################################
POC 05 — Flag 4 Captured
SSH login banner displaying the flag immediately on authentication — no shell commands needed.
Flag 4 SSH login banner / MOTD → admin@target.ine.local
Method: SSH login → MOTD / banner displayed automatically on connect
⚑ Technique Note
SSH Banners Are Always Worth Reading
/etc/motd and /etc/issue.net display to every authenticated SSH session automatically. In real environments these files frequently contain internal system information, configuration hints, or — as shown here — sensitive strings inadvertently left by administrators. Always read the full banner before issuing any commands. No further enumeration of the filesystem was required for this flag.

★ Key Takeaways
Lessons from Enumeration CTF 1
1. Always run -p-. FTP on port 5554 was completely invisible to a default scan. A full port scan is non-negotiable for any assessment — the time cost is always worth it.

2. Null-session SMB is the correct first step. It costs nothing and frequently reveals shares, usernames, and files outright before a single credential is needed.

3. Automate share iteration. A Bash loop over a wordlist is faster, more thorough, and more systematic than any manual approach. Build this reflex into SMB enumeration.

4. Port obscurity is not security. Hydra's -s flag attacks any port. Once identified, an FTP service on port 5554 is no harder to brute-force than one on port 21.

5. Read SSH banners carefully. MOTD content displays automatically and is frequently overlooked — it can contain flags, credentials, or system information without requiring any further enumeration.
